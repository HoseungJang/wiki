## 오프셋 기반 페이지네이션의 문제
항상 MySQL에서 페이지네이션에 offset을 사용하는 편이었는데, offset 값이 커질 수록 성능 저하가 심해진다는 문제점이 있다고 한다.

[이 글](https://www.eversql.com/faster-pagination-in-mysql-why-order-by-with-limit-and-offset-is-slow/)을 보면 데이터 수가 많아질 수록 offset을 사용하면 성능 저하가 심해지는걸 볼 수 있다.

## 커서 기반 페이지네이션
오프셋 기반 페이지네이션이 데이터를 몇 개나 뛰어넘을지에 대한 개념이라면, 커서 기반 페이지네이션은 어떤 데이터부터 가져올지에 집중한 개념이다.

아래와 같이 food 테이블에 데이터가 10개 있다고 가정해보자.

**food 테이블**

|id|name|
|---|---|
|1|김말이|
|2|떡볶이|
|3|김치|
|4|김밥|
|5|라면|
|6|미역국|
|7|라볶이|
|8|제육덮밥|
|9|고기볶음|
|10|김치찌개|

데이터를 4번째 행부터 5개씩 불러오고 싶다고 할 때, 오프셋, 커서 기반의 SQL query는 각각 아래와 같다.

- 오프셋 기반 페이지네이션
```
SELECT * FROM food ORDER BY id ASC OFFSET 3 LIMIT 5;
```

- 커서 기반 페이지네이션
```
SELECT * FROM food WHERE ID >= 4 ORDER BY id ASC LIMIT 5;
```

쿼리가 매우 간단하지만, 정렬 옵션 같은것을 추가하게 된다면 조금 더 복잡해질 듯 하다.

## 구현
typeorm의 repository과 express를 이용해서 커서 기반 페이지네이션을 구현해보았다.

우선 findByCursor 메소드를 만들어 주었다.

```typescript
public static async findByCursor(cursor: string) {
  return (await this.getRepository()).find({
    where: { id: MoreThanOrEqual(cursor) },
    order: { id: "ASC" },
    take: 15,
  });
}
```

위 코드는 결과적으로 아래의 쿼리를 생성한다.

```
SELECT * FROM post WHERE id >= (cursor) ORDER BY id ASC LIMIT 15;
```

그리고 간단히 express request handler를 만들어주었다.

```typescript
// GET /posts
async (req, res, next) => {
  try {
    const cursor = req.query.cursor as string;

    const result = await Post.findByCursor(cursor);
    const nextCursor =
      result.length === 15 ? result[result.length - 1].id + 1 : null;

    return res.status(200).json({
      result,
      nextCursor,
    });
  } catch (error) {
    return next(error);
  }
}
```

response body에서 다음 요청을 위한 cursor를 응답하고 있고, 클라이언트에선 그것을 사용하면 된다.
