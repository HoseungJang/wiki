## 3 way handshake

![](/네트워크/TCP/images/3-way-handshake.png)

3 way handshake는 TCP 연결을 수립하는 과정이다.

1. 송신측에서 연결 요청 패킷(SYN(X))을 보낸다.
2. 수신측에서 송신측의 연결 요청 패킷에 응답(ACK(X + 1))하고, 송신측에도 연결 요청 패킷(SYN(Y))를 보낸다.
3. 송신측에서 수신측의 연결 요청에 응답(ACK(Y + 1))함으로써 연결이 수립된다.

이 때, 각각의 연결 요청 패킷에는 임의의 난수가 사용되는데, 순차적인 수를 사용하면 기존의 connection에서 온 패킷과 혼동될 여지가 있어서 그렇다고 한다.

## 4 way handshake

![](/네트워크/TCP/images/4-way-handshake.png)

4 way handshake는 TCP 연결을 해제하는 과정이다. 송신측과 수신측 둘 중 한곳에서 먼저 시작하게 된다. 요청을 먼저 보낸 쪽을 client, 받은 쪽을 server라고 부르겠다.

1. client에서 연결 해제 요청(FIN)을 보낸다.
2. server에서 연결 해제 요청에 응답(ACK)하고, 이 때 client는 연결 해제를 기다리는 상태로 넘어간다.
3. server에서 아직 보내야할 데이터가 남은 경우, 남은 데이터를 모두 전송한 후 연결 해제 요청(FIN)을 client에 보낸다.
4. client가 연결 해제 요청에 응답(ACK)함으로써 TCP 연결이 완전히 해제되게 된다.

## 2 way handshake의 문제

![](/네트워크/TCP/images/2-way-handshake.png)

예를 들어, 만약 TCP 연결을 2 way로 수립했다면 어떻게 됬을까? 아래와 같은 예외 상황이 발생할 수 있다.

1. 송신측이 연결 요청을 보낸다.
2. 수신측이 연결 요청에 응답했지만, 송신측에서 timeout 등의 문제로 수신측의 응답을 인식하지 못하고 연결 요청을 다시 보낸다.
3. 수신측만 연결이 수립됬다고 판단하는 반쪽자리 커넥션이 생성된다.

문제는 4 way handshake에서도 마찬가지이다. 아래와 같은 상황이 벌어질 수 있다.

1. client가 연결 종료 요청을 보낸다.
2. server가 연결 종료 요청에 응답했지만, 아직 보낼 데이터가 남아있다.
3. client가 연결을 완전히 종료해 버려서 server의 남은 데이터가 모두 유실된다.

따라서 2 way handshake는 불완전하고, 3 way/4 way로 안전하게 연결을 수립/해제하는 것이다.
