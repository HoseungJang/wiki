React에서 렌더링할 때, 엘리먼트의 배열을 children으로 넘겨본 적이 있을 것이다.

이 때, 왜 key를 넘겨주는걸 권장하는 걸까? 그 이유는, 배열 요소의 추가/변경/삭제에 효율적으로 대응하기 위해서이다.

예시를 들어보겠다. 아래와 같이 데이터를 렌더링한다고 가정해보자.

```typescript
const items = [
  { id: 0, name: "A" },
  { id: 1, name: "B" },
  { id: 2, name: "C" },
];
```

```tsx
<ul>
  {items.map((item) => <li key={item.id}>{item.name}</li>)}
</ul>
```

여기서 새로운 요소가 추가되어 배열이 변경되면 어떻게 될까? 배열의 맨 앞에 새로운 요소가 추가되었다고 해보자.

```typescript
const items = [
  { id: 3, name: "D" },
  { id: 0, name: "A" },
  { id: 1, name: "B" },
  { id: 2, name: "C" },
];
```

결과적으로 아래와 같이 컴포넌트 트리가 생성될 것이다.

```tsx
<ul>
  <li key={3}>D</li>
  <li key={0}>A</li>
  <li key={1}>B</li>
  <li key={2}>C</li>
</ul>
```

이 때 React는 key값을 통해 '3' key를 가진 엘리먼트만 추가되었고, 나머지 0, 1, 2는 이동만 해주면 된다는 것을 파악하여 효율적으로 DOM을 업데이트한다.

## 배열의 인덱스를 키 값으로 사용
배열의 인덱스를 키 값으로 사용하는 것은, 요소의 순서가 달라졌을 때 문제가 될 수 있다.

맨 뒤에 요소가 추가되는 등의 상황은 문제가 되지 않지만, 중간에 있던 요소가 삭제되거나 맨 앞에 요소가 추가되는 등의 경우는 기존 요소의 key 값에 변화가 일어나게 된다.

예를 들어, 맨 앞에 요소가 추가되면 기존 요소들의 key값은 모두 +1씩 증가하게 된다.

React는 key값에 의존하여 리렌더링을 진행하므로, 비효율적인걸 넘어서 렌더링 결과 자체가 이상해질 수도 있다. 따라서 배열의 인덱스는 키 값으로 함부로 쓰지 않는 것이 좋다.
